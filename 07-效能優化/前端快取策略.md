# 前端快取策略

> 前端快取的目的不是單純讓資料「留在本地」，而是減少重複請求、加快載入速度、降低伺服器壓力，並在更新時保持可控。

---

## 為什麼要懂快取

如果你只知道「重新整理就會抓最新資料」，那在實務上很容易遇到這些問題：

- API 明明更新了，畫面還是舊資料
- JS/CSS 部署後，使用者還載到舊版
- 圖片更新了，但 CDN 還吐舊圖
- 離線支援做了，結果永遠不更新

快取的難點不只是「快」，而是：

- 什麼時候該用舊資料
- 什麼時候一定要更新
- 更新後怎麼讓使用者拿到新版本

---

## 常見快取層級

### 1. 記憶體快取

資料放在執行中的記憶體裡，速度最快，但頁面關掉通常就沒了。

常見例子：

- 變數
- React state
- [[React Query]] cache

### 2. 瀏覽器快取

瀏覽器會根據 HTTP response headers 決定資源能不能直接重用。

常見對象：

- JS
- CSS
- 圖片
- API response

### 3. Service Worker 快取

由 Service Worker 自己攔截請求並決定回什麼。

適合：

- PWA
- 離線頁面
- 自訂資源更新策略

---

## HTTP Cache 核心觀念

### 強快取

如果資源還在有效期內，瀏覽器直接用本地快取，不會問伺服器。

常見 header：

```http
Cache-Control: max-age=31536000
```

### 協商快取

快取過期後，瀏覽器會先問伺服器：

「我手上這份還能不能繼續用？」

常見 header：

- `ETag`
- `If-None-Match`
- `Last-Modified`
- `If-Modified-Since`

如果沒變，伺服器回 `304 Not Modified`。

## 靜態資源怎麼做

最常見的穩定做法：

1. 檔名帶 hash
2. 設長時間快取
3. 每次部署產生新檔名

例如：

- `app.abc123.js`
- `styles.xyz999.css`

這樣好處是：

- 舊檔可以長時間快取
- 新版因為檔名不同，使用者會拿到新資源

這是 [[Vite(主流)]]、Webpack 這類建置工具常見的做法。

---

## API 資料快取怎麼想

API 通常不能像靜態資源一樣無腦快取超久，因為資料會變。

常見策略：

- 短時間快取
- stale-while-revalidate
- 手動失效
- 標籤式失效

例如在 [[React Query]] 中，你會常看到：

- `staleTime`
- `cacheTime` 或 gc time
- `invalidateQueries`

這類工具其實就是在幫你管理資料什麼時候算舊、什麼時候要重抓。

---

## Service Worker 的常見策略

### Cache First

先回快取，沒有再打網路。

適合：

- 字型
- logo
- 不常變的圖片

### Network First

先打網路，失敗才回快取。

適合：

- 即時資料頁面
- 新聞列表
- 需要最新資訊的 API

### Stale While Revalidate

先回舊快取讓畫面快出來，再背景更新。

適合：

- 可接受短暫舊資料的頁面

---

## 常見錯誤

### 把 HTML 也快取太久

如果 HTML 本身快取太久，使用者可能永遠拿到舊版入口檔。

### 只有清 CDN，沒處理檔名版本

這通常不夠穩。最好還是讓靜態資源檔名帶版本或 hash。

### 資料已更新，但 UI 沒有失效機制

像送出表單後沒 `invalidateQueries`，畫面就會一直顯示舊資料。

---

## 實務建議

### 靜態檔案

- 檔名加 hash
- 長快取
- HTML 短快取或不快取

### API 資料

- 根據資料變動頻率決定 stale time
- 重要操作後主動失效
- 不要所有資料都設超長 cache

### PWA

- 明確定義更新策略
- 避免把所有請求都一律 cache first

---

## 面試怎麼回答

> 前端快取可以分成記憶體快取、瀏覽器 HTTP Cache 跟 Service Worker 快取。靜態資源通常用 hash 檔名配長快取，API 資料則要根據更新頻率做協商或手動失效。快取的重點不是只求快，而是快取與更新要同時可控。
