# 元件測試實戰

> 測試的對象是「使用者行為」，不是「實作細節」。這是 React Testing Library 的核心哲學。

→ [[前端工程師]]

---

## Summary

元件測試（Component Testing）驗證 React 元件在不同 props、互動、狀態變化下的渲染行為。
主流工具是 **React Testing Library（RTL）** 搭配 **Jest** 或 **Vitest**。

---

## Plain Explanation

元件測試就是模擬使用者的角度來測試：
- 頁面顯示了什麼文字？
- 按下這個按鈕後畫面變成什麼？
- 表單送出後有沒有呼叫正確的函式？

不關心元件內部用了幾個 `useState`，只關心「看到的」和「做到的」。

---

## 核心原則：不測實作細節

```tsx
// ❌ 測試實作細節（fragile，重構就壞）
expect(component.state('count')).toBe(1);

// ✅ 測試使用者看到的（stable，重構不壞）
expect(screen.getByText('目前數量：1')).toBeInTheDocument();
```

---

## 基本範例

### 元件
```tsx
// Counter.tsx
export function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <p>目前數量：{count}</p>
      <button onClick={() => setCount(c => c + 1)}>+1</button>
    </div>
  );
}
```

### 測試
```tsx
import { render, screen, fireEvent } from '@testing-library/react';
import { Counter } from './Counter';

test('點擊按鈕後數字增加', () => {
  render(<Counter />);

  expect(screen.getByText('目前數量：0')).toBeInTheDocument();

  fireEvent.click(screen.getByRole('button', { name: '+1' }));

  expect(screen.getByText('目前數量：1')).toBeInTheDocument();
});
```

## 查詢優先順序（Query Priority）

RTL 鼓勵用「使用者能感知的方式」查詢元素：

| 優先級 | 方法 | 適用情境 |
|--------|------|---------|
| 最高 | `getByRole` | 按鈕、輸入框、標題等有語意的元素 |
| 高 | `getByLabelText` | 表單欄位（對應 label） |
| 高 | `getByPlaceholderText` | 只有 placeholder 的輸入框 |
| 中 | `getByText` | 純文字內容 |
| 低 | `getByTestId` | 最後手段，加 `data-testid` 屬性 |

## 測試非同步行為

```tsx
import { render, screen, waitFor } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

test('送出表單後顯示成功訊息', async () => {
  render(<LoginForm />);

  await userEvent.type(screen.getByLabelText('帳號'), 'user@example.com');
  await userEvent.type(screen.getByLabelText('密碼'), 'password123');
  await userEvent.click(screen.getByRole('button', { name: '登入' }));

  // 等待非同步結果
  await waitFor(() => {
    expect(screen.getByText('登入成功')).toBeInTheDocument();
  });
});
```

## Mock 外部依賴

```tsx
// Mock API 呼叫
jest.mock('../api/user', () => ({
  fetchUser: jest.fn().mockResolvedValue({ name: 'Eric', role: 'admin' }),
}));

test('載入後顯示使用者名稱', async () => {
  render(<UserProfile userId="123" />);
  expect(await screen.findByText('Eric')).toBeInTheDocument();
});
```

## 什麼該測、什麼不該測

| 該測 | 不該測 |
|------|--------|
| 元件的渲染輸出 | 元件的 state 數值 |
| 使用者互動後的 UI 變化 | 內部 function 有沒有被呼叫 |
| 條件渲染（loading / error / success） | CSS class name |
| 表單驗證錯誤訊息 | 元件的 ref 值 |

---

## 常見錯誤

- **用 `getBy` 查詢非同步元素**：應改用 `findBy`（內建 await）
- **直接測試 `useState` 的值**：測試實作細節，應改為測試 UI 輸出
- **`act()` 警告**：通常是因為 state 更新發生在 render 之後但沒有包在 `waitFor` / `findBy` 裡
