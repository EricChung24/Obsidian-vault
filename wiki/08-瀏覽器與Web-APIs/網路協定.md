## 網路分層模型

```
OSI 七層                    TCP/IP 四層
─────────────────────       ──────────────────
7. 應用層 (HTTP, DNS)   ─┐
6. 表示層               ─┤→ 應用層 (HTTP, HTTPS, DNS, FTP)
5. 會話層               ─┘
4. 傳輸層 (TCP, UDP)    ──→ 傳輸層 (TCP, UDP)
3. 網路層 (IP)          ──→ 網路層 (IP, ICMP)
2. 資料鏈路層           ─┐
1. 實體層               ─┘→ 網路存取層 (Ethernet, Wi-Fi)
```

## TCP vs UDP

| 特性 | TCP | UDP |
|------|-----|-----|
| 連線 | 需要建立連線 | 無連線 |
| 可靠性 | ✅ 保證交付、順序、完整性 | ❌ 不保證 |
| 速度 | 較慢（握手、確認）| 較快 |
| 使用場景 | HTTP、HTTPS、SSH、FTP | 直播、遊戲、DNS |
| Header 大小 | 20-60 bytes | 8 bytes |

## TCP 三次握手（建立連線）

```
客戶端                          伺服器
   │                              │
   │──── SYN（seq=x）────────────→│  客戶端：「我想連線，序號是 x」
   │                              │
   │←─── SYN-ACK（seq=y, ack=x+1）│  伺服器：「收到，序號是 y，等你的 x+1」
   │                              │
   │──── ACK（ack=y+1）──────────→│  客戶端：「確認，你的 y+1」
   │                              │
   │         連線建立              │
```

**為什麼需要三次？**
- 兩次無法確認雙方的發送和接收能力都正常
- 三次可以確認：客戶端能收發、伺服器能收發

## TCP 四次揮手（斷開連線）

```
客戶端                          伺服器
   │                              │
   │──── FIN（seq=u）────────────→│  客戶端：「我發完了，想關閉」
   │                              │
   │←─── ACK（ack=u+1）──────────│  伺服器：「收到，但我還有資料要傳」
   │                              │  （此時進入 CLOSE_WAIT，繼續傳資料）
   │                              │
   │←─── FIN（seq=v）────────────│  伺服器：「我也傳完了」
   │                              │
   │──── ACK（ack=v+1）──────────→│  客戶端：「好，連線關閉」
   │                              │  （客戶端等待 2MSL 後才真正關閉）
   │         連線關閉              │
```

**為什麼需要四次？（不是三次？）**
- 伺服器收到 FIN 後，可能還有資料要傳
- 需要等待資料傳完，才能發送自己的 FIN
- 所以 ACK 和 FIN 分兩次發送

**TIME_WAIT 等待 2MSL 的原因：**
- 確保最後的 ACK 到達伺服器（如果丟失，伺服器會重發 FIN）
- 讓舊連線的資料包在網路中消失，避免影響新連線

## HTTPS 握手（TLS 1.3）

```
客戶端                              伺服器
   │                                  │
   │──── Client Hello ───────────────→│
   │     （支援的加密套件、隨機數 C）     │
   │                                  │
   │←─── Server Hello ───────────────│
   │     （選定加密套件、隨機數 S、憑證）  │
   │                                  │
   │  [客戶端驗證憑證、生成 Pre-Master]  │
   │                                  │
   │──── Change Cipher / Finished ──→│
   │     （加密通訊開始）                 │
   │                                  │
   │←─── Finished ───────────────────│
   │                                  │
   │         加密連線建立               │
```

**TLS 1.3 改進：**
- 只需 1 個 RTT（1.2 需要 2 個 RTT）
- 支援 0-RTT 恢復連線（重連更快）

## DNS 解析過程

```
瀏覽器輸入 www.example.com

1. 瀏覽器快取  ── 有? → 直接使用
                       ↓ 沒有
2. 系統 hosts  ── 有? → 直接使用（/etc/hosts）
                       ↓ 沒有
3. 本機 DNS 快取
                       ↓ 沒有
4. 遞迴解析器（Router / ISP DNS）
                       ↓ 快取未命中
5. 根域名伺服器  → 回傳 .com 的 TLD 伺服器地址
                       ↓
6. TLD 伺服器（.com）→ 回傳 example.com 的權威伺服器
                       ↓
7. 權威 DNS 伺服器  → 回傳 www.example.com 的 IP
                       ↓
8. 快取結果（依據 TTL），連線到 IP
```

## 輸入 URL 到頁面顯示（完整流程）

```
1. URL 解析
   → 判斷是否為合法 URL，分析協定/域名/路徑/參數

2. DNS 解析
   → 取得伺服器 IP（見上方）

3. 建立 TCP 連線
   → 三次握手

4. TLS 握手（HTTPS）
   → 協商加密方式、驗證憑證

5. 發送 HTTP 請求
   GET / HTTP/2
   Host: example.com
   Cookie: session=abc

6. 伺服器處理並回應
   HTTP/2 200 OK
   Content-Type: text/html
   Cache-Control: no-cache

7. 瀏覽器接收 HTML 並開始解析
   → 邊下載邊解析（流式處理）

8. 建構 DOM + CSSOM
   → 遇到 <script> 可能暫停解析（除非 defer/async）

9. 合成 Render Tree、Layout、Paint、Composite
   → 頁面顯示

10. 載入子資源（圖片、JS、字體...）
    → 觸發更多 HTTP 請求

11. JS 執行
    → 互動事件綁定、資料請求等

12. 頁面完全載入（load 事件觸發）
```

## HTTP/1.1、HTTP/2、HTTP/3 比較

| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|---------|--------|--------|
| 底層協議 | TCP | TCP | QUIC（UDP）|
| 多工 | ❌（一個連線一個請求）| ✅ | ✅ |
| Header 壓縮 | ❌ | ✅ HPACK | ✅ QPACK |
| 隊頭阻塞 | ❌ 有 | ❌ 仍有（TCP 層）| ✅ 解決 |
| Server Push | ❌ | ✅ | ✅ |
| 連線遷移 | ❌ | ❌ | ✅（切換網路不斷線）|

## 常見面試考點

### Q: 為什麼 HTTPS 比 HTTP 安全？

1. **加密**（Encryption）：資料在傳輸中被加密，竊聽者無法讀取
2. **驗證**（Authentication）：憑證驗證伺服器身份，防止中間人攻擊
3. **完整性**（Integrity）：MAC 確保資料未被篡改

### Q: GET vs POST 的差別？

| | GET | POST |
|--|-----|------|
| 資料位置 | URL 查詢字串 | Request Body |
| 長度限制 | URL 有限制（~2000 chars）| 無限制 |
| 快取 | 可快取 | 不快取 |
| 安全性 | 資料暴露在 URL | 相對安全 |
| 冪等 | ✅ | ❌ |
| 用途 | 讀取資源 | 建立/提交資料 |

### Q: 什麼是 CDN？

Content Delivery Network，全球分散節點的快取伺服器。
- 用戶連到最近的節點（降低延遲）
- 靜態資源快取（JS/CSS/圖片）
- DDoS 防護、SSL 終止
