# 狀態管理比較：Redux vs Zustand vs Pinia

> 沒有最好的狀態管理工具，只有最適合當下規模與團隊的選擇。

→ [[前端工程師]]

---

## 快速選型結論

| 情境 | 建議 |
|------|------|
| React 小型專案 / 個人專案 | [[Zustand]] |
| React 大型團隊 / 有 Redux 歷史包袱 | [[Redux]]（搭配 Redux Toolkit） |
| Vue 專案（任何規模） | [[Pinia]] |
| 需要嚴格的單向資料流與 debug | [[Redux]] |
| 想要最少 boilerplate | [[Zustand]] 或 [[Pinia]] |

---

## 三者定位

### Redux
- 老牌 Flux 架構實作，強調**單向資料流**與**可預測性**
- 現代用法是 **Redux Toolkit（RTK）**，大幅簡化 boilerplate
- DevTools 最成熟，適合需要 time-travel debugging 的大型專案
- 學習曲線較高，概念多（store / slice / reducer / selector / thunk）

### Zustand
- React 生態中目前最受歡迎的輕量方案
- API 極簡：一個 `create()` 搞定 store，無需 Provider
- 支援 middleware（immer、devtools、persist）
- 沒有強制架構，靈活但也容易寫亂

### Pinia
- Vue 3 官方推薦的狀態管理（Vuex 的後繼者）
- 使用 Composition API 風格，與 Vue 3 無縫整合
- TypeScript 支援優秀
- 支援 Options API 和 Composition API 兩種寫法

## 概念對比表

| 概念 | Redux | Zustand | Pinia |
|------|-------|---------|-------|
| 框架 | React | React | Vue |
| 資料流 | 單向（嚴格） | 單向（寬鬆） | 單向（寬鬆） |
| 核心概念 | store / slice / reducer | store / action | store / state / action / getter |
| 非同步 | thunk / saga / RTK Query | 直接在 action 寫 async | 直接在 action 寫 async |
| DevTools | 最完整 | 支援 | 支援 |
| Bundle 大小 | 較大（但 RTK 合理） | 極小（~1KB） | 小 |
| Provider 需求 | 需要 `<Provider>` | 不需要 | 不需要 |

---

## 何時不需要狀態管理工具

以下情況用 React 內建的 `useState` / `useContext` 就夠了：
- 狀態只在少數幾個元件之間共用
- 資料不需要持久化或同步到 server
- 專案規模小、複雜度低

→ 加工具之前先問：「這個狀態真的需要全域嗎？」

---

## 常見誤解

- **誤解**：Redux 很難用。→ 現代 Redux Toolkit 已大幅簡化，不用再手寫 action type 和 reducer。
- **誤解**：一個 React 專案只能選一種。→ 技術上可以並存，但實務上應該統一。
- **誤解**：Zustand 不適合大型專案。→ 只要設計良好的 store 切分，Zustand 也能支撐大型專案。
